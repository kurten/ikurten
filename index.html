Redis网络模块记录

2014-07-25 | Comments

redis实现了一个非常小巧的网络IO库，外部调用接口主要由ae.h头文件定义，以及ae.c做实现。ae.c内部调用的对应平台的不同实现采用编译时宏定义include的方式加载。主要有ae_epoll.c(linux epoll)、ae_evport.c(Solaris)、ae_kqueue.c(Kqueue)、ae_select.c(select)。

初始化网络监听文件句柄并且添加至eventloop(redis.c/initServer-第1540行调用了anet.c/anetTcpServer函数生成tcp文件句柄,第1550行是基于unix socket文件的句柄。redis.c第1624行以及第1626行分别是创建aeFileEvent并添加对应文件句柄至eventloop的操作，其中分别把networking.c/acceptTcpHandler和networking.c/acceptUnixHandler添加作为rfileProc的处理函数)。在main函数中，先设置eventloop结构的beforeSleep函数在aeProcessEvents函数执行前执行，aeProcessEvents执行会计算租塞获取events的时间，取到对应的event然后执行相应的rfileProc或者wfileProc。下面以tcp为例子做说明。

当服务端收到连接请求时，前面anetTcpServer生成的事件监听会被触发，acceptTcpHandler被调用然后执行acceptCommonHandler同时生成一个终端redisClient对象，并被加入只服务端的终端列表中，具体操作在networking.c/createClient函数中，同时将对应的文件句柄注册到eventloop中，对应的回掉函数为networking.c/readQueryFromClient。

当该终端接收到客户端发送的数据时，aeProcessEvents中会触发该终端对应文件句柄的注册事件的回掉函数readQueryFromClient，第一次读取最大16k的数据缓存至querybuf，然后执行processInputBuffer，在processInputBuffer函数中根据reqtype情况判断是否为REDIS_REQ_MULTIBULK(默认为REDIS_REQ_INLINE)。判断依据为querybuf中首字节为’*’(具体请查看redis protocol)。REDIS_REQ_INLINE对应处理函数processInlineBuffer，REDIS_REQ_MULTIBULK对应处理函数processMultibulkBuffer。

对于processInlineBuffer，如果处理querybuf时未找到’\r\n’符号，则出错，退出processInputBuffer的while循环，并且调用resetClient重置终端的相应参数。processMultibulkBuffer稍稍复杂，先判断’\r’然后再判断’\n’，出错则返回。这里两个函数其实是具体的协议解析，结合redis protocol查看就事半功倍。解析完协议串，然后会存储至clientObject中的argc(数组长度)和argv(redisObject类型为REDIS_STRING数组指针)。

然后再redis.c/processCommand中处理具体的命令操作。处理完成后调用resetClient重置终端的相应参数。如果，处理命令出错或者命令处理完成之后需要发送处理的结果，则调用’networking.c/addReply*’系列函数，该系列函数首先都会执行networking.c/prepareClientToWrite，该函数会注册写的事件监听至eventloop，对应的回掉函数即wfileProc指向sendReplyToClient函数。

最近由于在看《UNIX环境高级编程》，于是想起来redis的源码以前没怎么深入，然后就结合着Redis 设计与实现（第一版）看看redis-2.6的源码，发觉缺少了对于网络通信模块的叙述，于是便有了此文。简单介绍网络相关的收发逻辑以及对应处理函数，未深入介绍内部逻辑，后续有空在做深入介绍吧。

IM开发日志(2)

2014-04-23 | Comments

bug

今天碰到了一个比较诡异的问题，我开发的IM协议的android sdk突然出现了OOM，导致整个程序进程崩溃，无法重新启动。通过异常日志定位，排查下来，主要问题是出在接收解析包头的时候，解析出来的后续数据长度超过了可分配的当前内存大小，导致整个程序OOM崩溃了。

最初设计通讯协议的时候，没有考虑到TCP通讯在网络切换等状况下，会导致通讯包的不完整（即粘包、丢包）的问题，没有设计包的结束符，所以处理这个问题还是有些麻烦的。之前大致考虑了一下，给了一个零时的修补方案，取到的包头，计算后续包长度如果超过一个临界值，则直接丢弃这个包头，继续尝试获取包头。现在想想，可以尝试获取指定长度的一串包，直接判断包内容，如果内容中出现特定的关键key，则从头尾进行搜索尝试读取其中完整的包，否则丢弃。

当然，如果设计了通讯包的开头符号和结束符号，那么完全不用这么麻烦，直接尝试读取开头符号和结尾符号去判断是否是完整包就可以了，如果不完整，缓存下来，等待下一个包拼接解析就可以了。

关于web登录

现阶段，IM的web这块采用socket.io来实现的，有个问题原先没有考虑，就是一个用户如果在一个浏览器中开多个tab页面，如果保证用户在线状态。现在的实现是一个用户，只能开一个tab页面，否则就会被强制退出。然后我想，可以用一个cookie去标识用户登录的浏览器是否是同一个，如果是同一个就不强制退出，否则就强制退出前面登录的。原先以为socket.io和express结合应该可以实现cookie访问，但是尝试了一下，有查看了一下socket.io的源码，发觉返回的headers中没有Set-Cookie这个头，于是只好自己在原先的用户登录体系中加入了cookie标识，在IM登录时把这个cookie标识传送过来做比对。这样应该就可以实现同个浏览器多tab登录了。

关于IM新的设计思路

目前，主要细分了session服务、router服务、store服务以及offline服务，每个服务都是一个nodejs进程在跑，各个服务直接通过message queue做rpc，所以每个服务只要管理对应的inbox队列就可以了，队列可以多实例容错，各个服务也可以多实例。现在有个单点服务就是所有用户在线状态是存储于router服务中的，session服务只存储连接在自己上的用户。

为了保证消息能够一接收就持久化下来，采用了memcacheq，基于memcached协议的消息队列，简单、轻量，但是不支持推送。当然，所有服务rpc都是经过该队列服务，这就导致了一个rpc消息有两次磁盘io的开销，而且中间还有get循环的间隙，接下去打算用redis的pub/sub来替换。同时，有了redis可以把用户在线状态存储在redis里，这就去掉了所有用户在线状态只有router服务知晓的这个单点，对于后续水平扩容更方便了。

IM开发日志(1)

2014-03-25 | Comments

最近忙着业务上面来回修改问题以及对应的bug，还有就是我们自有协议的即时通讯系统的重构，事情一多，就犯懒，懒得写东西。这两天，业务上的东西暂告一段落，专注重构通讯系统，于是就写篇文章作为记录。

Version 1.0

第一版架构非常简单，设计有3中服务类型，session管理服务，router服务，以及monitor服务。其中session管理服务主要管理web登录的session和mobile客户端tcp方式登录的session。基于web的session管理服务还需要实现支持消息推送的接口，这些接口给web产品的服务端做业务的消息推送使用。router服务统一作为消息路由（消息存储、消息转发、消息群发）。monitor服务主要负责监控集群的状态以及相应的集群管理功能（重启、停止等）。

当然，还有许多问题没有考虑，比如消息延迟、集群增多后消息分发顺序、用户在线离线状态是否共享等等。

由于业务上面变更太多，web产品上面原先的需求和现在的需求变更太大，导致原有版本无法更好的进行水平扩展，所以我进行了重构。

Version 2.0

这一版session管理服务不变，引入消息队列服务作为消息中转服务，保证消息的分发顺序，同时增强水平扩展能力，同样有router服务监听消息分发队列做消息分发。还可以根据业务变更增加相应的消息路由规则。即增加消息队列以及对应的消息处理映射至router服务。消息处理完成之后再放入分发队列由router服务处理分发。这样，不管后续业务如何变更，只要增加消息处理的服务和映射关系就可以了。

当然，额外要做的就是集群机器扩展的通知（可以自己定义一个规则，通过队列通知各个服务更新配置信息即可）。

集群的监控也需要重新实现。虽然中间环节变多，但是增强水平扩展能力，不导致业务变更而再去变更通讯系统的实现逻辑，这也是一种取舍。

该系统目前使用node.js实现，所以对于一个服务其实就是一个process，在linux下使用pm2管理，消息队列采用memcacheq，因为以前用过，也比较熟悉所以使用它作为队列服务。memcacheq是pull模式的，考虑后续重新实现一个push模式的消息队列服务（基于memcached协议），这样的话，对于异步的程序逻辑，更合适一点，同时剥离掉memcacheq的bdb队列依赖，自己实现一个队列存储的模式然后使用libuv作为网络库。这些当然后续有时间去做了。

开发日志(with node.js)

2014-03-01 | Comments

node.js开发

最近的项目一直在用node.js开发，包括web端、移动端的api实现，以及后台消息系统都是用node.js。web端和api实现都是基于express这个框架基础上去做包装实现的，逐渐构建了一个类似于rails的框架。开发人员只要注重实现controller和view以及前台页面交互就可以了，不需要考虑其他模块载入，封装什么的内容。controller实现只要遵照一种约定即可。

目标需求大致罗列如下：

  • 模块自动载入(controller、middleware、config等)
  • response.render页面时公共输出内容加载输出
  • web app的安全配置(copy了lusca)
  • 前端模板引擎要求可以使用cdn加速
  • 自动化编译发布
  • 部署方便，快速

于是采用的技术选型为:

  • nodemon(开发环境使用)
  • bower(前端依赖安装)
  • grunt(自动化打包编译发布)
  • pm2(正式环境使用)
  • async(用来解决回掉函数太多的问题，比如sql查询的回掉)
  • dustjs-linkedin(可以编译为js，用于cdn加速)
  • 自己实现了模块载入工具对象以及模板渲染输出时的hook
  • 前端基础样式UI库用bootstrap和compass，方便编写

controller实现的约定


/**
 * 遍历文件夹，获取并且绑定controller中导出的方法
 * 仅支持多级目录
 * controller导出的模式为
 * module.exports = {
 *     hello:{method:'get', path:'/hello', fun:function (req, res) { res.send('hello world'); }}
 * }
 * -------- or --------
 * exports.method = 'get';
 * exports.path = '/hello';
 * exports.fun = function (req, res) {
 *     res.send('hello world');  
 * }
 */

middleware实现的约定


/**
 * 载入自定义中间件 例如 auth等
 * 导出模式(支持单个中间件中多个导出函数以及多级目录)
 * exports.path = "/hello";   //可选选项没有默认为'*'
 * exports.auth = function(req, res, next) {
 *     next();  
 * }
 */

render hooks实现的约定


/**
 * 载入自定义hooks 例如 hookForAll.js等
 * 导出模式(支持多级目录)
 * exports.path = '*';
 * exports.fun = function (options, req, res, cb) {
 *     cb && cb(options);
 * };
 */
